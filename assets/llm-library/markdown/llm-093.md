## Abstract + Introduction
### 背景
- 盲目认为智能体的数量越多, 性能就会越好, 并且声称MAS优于SAS
- 静态评估具有误导性: 大量的Agent评估在非Agentic的静态基准(如GSM8K, HumanEval) 上, 这些任务往往能够带来性能提升, 但这并不能代表正式的动态环境交互能力
### 挑战
- 缺乏原则性的框架: 研究人员依赖启发式而非原则性设计来选择框架, 没能理性选择MAS还是SAS
- 成本与效益不明: 缺乏一个定量的框架来预测多智能体协作带来的协调开销何时能够超过其带来的收益
### 贡献
>[!核心论点] 
> - 智能体系统的性能并非仅仅随着智能体数量增长而单调增长, 而是遵循着智能体数量、协调结构、模型能力和任务属性相互作用而决定的Scaling Laws
> - 刻画了基础模型能力与智能体性能的非线性关系, 虽然更高的能力带来加速回报, 但这些收益并非自动获得, 他们严格依赖于架构对其. 若缺乏正确的协同结构, 基础能力的提升往往会被协同开销所抵消

### 智能体任务的定义
- 与外部环境进行持续多步交互
- 在部分可观测条件下进行迭代式信息搜集
- 基于环境反馈进行适应性策略优化

### 三种模式
#### 工具密集型任务
- 遭受多智能体协调开销的影响, 并且随着环境复杂度提升而降低效率
#### 单智能体成功率超过45%场景
- 增加智能体会带来负收益, 因为协调成本超过了收益递减的威力
#### 架构依赖的误差放大
- 成功的关键在于架构与任务是否协调, 而不在于智能体数量
- 最优架构不是一成不变的
	- 去中心化协调有利于并行搜索高熵环境(网络搜索), 而在所有智能体需要顺序执行任务则带来性能下降(规划), 因为在固定思考开销下, 协商会占用推理budget

### MAS与SAS的区分定义
![[Pasted image 20260209203004.png]]
- SAS
	- 所有的感知, 规划与行动均发生在同一Agent内部
- MAS
	- 他们通过结构化消息传递, 共享内存或协调机制进行通信
- MAS拓扑结构分类
	- 独立系统: 聚合孤立输出
	- 去中性化系统: 进行P2P交换
	- 中心化系统: 通过协调器进行路由
	- 混合系统: 结合了层级控制和横向通信

## Agent Systems and Tasks
### Define
- 一个系统 $S$ 为一个四元组 $(A, E, C, \Omega)$:
	- **$A$ (Agents):** 智能体集合 $\{a_1, ..., a_n\}$
	- **$E$ (Environment):** 共享的环境
	- **$C$ (Topology):** 通信拓扑结构
	- **$\Omega$ (Orchestration):** 编排策略（怎么做决定，比如投票还是独裁）
- 单个智能体的定义: 一个智能体 $a_i$ 被定义为 $S_i = (\Phi_i, \mathcal{A}_i, M_i, \pi_i)$
	- **$\Phi_i$ (Reasoning Policy):**  LLM 本身
	- **$\mathcal{A}_i$ (Action Space):** 包含工具使用的动作空间
	- **$M_i$ (Memory):** 这是**记忆**，存储历史上下文
	- **$\pi_i$ (Decision Function):** 这是**决策逻辑**，将观察到的历史映射到下一个动作
### 实验控制
- 作者将SAS和MAS的token预算都控制在同一水平
### 实验研究的核心问题
- 哪些因素决定了Agent System的性能
- 何种条件下, 智能体之间的协调会提神或降低Agent System的表现
- 能否从可测量的任务属性中推到出定量的缩放原则, 以预测给定任务的最佳智能体架构

## Exp
### 基准测试
- Workbench: 涉及确定性的代码执行和工具使用，具有客观的通过/失败标准,性能变异性较低
- Finance Agent: 需要多步定量推理和风险评估
- PlanCraft: 在约束条件下进行的针对Minecraft的时空规划任务
- BrowseComp-Plus: 动态网页导航、信息提取和跨页面综合 。该基准测试表现出最高的性能变异性

### 公平性控制
- 为了保证公平, MAS和SAS的迭代次数是匹配的, MAS获得更多的并行推理机会, 而SAS则获得更深的推理深度

### Agent架构与复杂度
- 复杂度量化
	- **协调复杂度**这一指标通过通信开销来量化:每个任务所需智能体间交换信息总次数, 实验测量值从 SAS 的 0% 到 Hybrid 的 515% 不等
### 指标
- **主要指标：** 任务成功率/准确率
- 次要/过程指标
	- **事实错误率 ($E$):** 通过特定领域的验证器测量
	- **信息增益 ($\Delta I$):** 比较协调前后的不确定性代理变量
	- **Token 重叠结构:** 分析代理理由中的 Token 是唯一的、共享的还是矛盾的。
	- **效率指标:** 包括每 1000 个 Token 的成功率等

	- 如果任务是串行的, 如PlanCraft, 那么MAS反而会拖累任务的表现
	- 四个基准上, MAS体现出来高方差的表现, 证明MAS并非提供普适性的优势, 而是在特定领域的权衡关系

###  机制解析
- 作者发现, 领域复杂度是MAS优势的显著负调节因子
- 当我们固定token budget
	- 低/中复杂度场景
		- 单智能体就能够很好的完成任务
		- 盈余资源用于通信消耗
		- 通信合成了局部解, 带来了巨大的性能提升
	- 高复杂度/强顺序场景
		- 单agent在状态追踪和推理思考就已经消耗绝大部分的token budget了
		- 导致用于通信的Token变得捉襟见肘
		- 恶性循环: 为了省token, 压缩了信息质量, 或者因为推理能力被分散了导致错误, 这解释了在PlanCraft的不良表现
- 信息增益验证
	- 结构化领域: 信息增益与成功率强相关, 智能体的交流确实带来了信息增益
	- 顺序领域: 相关性极弱, 智能体虽然在说话, 但却是无效沟通

### 作者的预测模型
- 工具越多的任务, 多智能体协作的效率惩罚越严重(这点似乎和Anthropic的观点不同)
	- 作者认为MAS会因为额外的协调税降低运行效率
- 能力上限
	- 经验上, 如果单智能体的成功率高于45%, MAS带来的效用抵不过成本
- 架构依赖的错误级联
	- Independent: 错误放大因子 $A_e = 17.2$。因为没有通信，一个 Agent 错了就全错了，且没人纠正
	- Centralized: 错误放大因子 $A_e = 4.4$。Orchestrator充当了“看门人”，拦截并修正了部分错误

### 架构选择建议
- **规划类任务 (Planning Tasks):**
    - 特征：工具少 ($T=4$)，基线尚可 ($P_{SA}=0.57$)。
    - 推荐：**Single-Agent (SAS)**。
    - 理由：触发了“能力天花板定律”，且不需要 MAS 来处理复杂工具。
- **分析类任务 (Analysis Tasks):**
    - 特征：工具中等 ($T=5$)，基线较难 ($P_{SA}=0.35$)。
    - 推荐：**Centralized MAS**。
    - 理由：基线低，有提升空间；Centralized 架构能有效控制错误放大 ($A_e=4.4$)。
- **工具密集型任务 (Tool-heavy Tasks):**
    - 特征：工具极多 ($T=16$)，基线高 ($P_{SA}=0.63$)。
    - 推荐：**Decentralized MAS**。
    - 理由：虽然开销巨大，但去中心化架构具有最高的**冗余度 ($R$)** 和并行能力，能够抵消部分效率损失。
