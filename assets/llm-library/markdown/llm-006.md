## Abstract + Introduction
### 背景
- 主导范式: 大多数LLM应用严重依赖上下文适应, 通过context来引导模型行为, 而不是直接修改模型权重
- 主流方法1: 提示词优化
	- 通过测试和反思, 得到最佳的静态提示词
- 主流方法2: 自适应内存
	- 运行时动态建立一个知识库, 让Agent能够从过去的成功或者失败学习
### 挑战
- 简洁性偏差
	- 许多系统prompt被设计为简洁和通用指令
	- 这种简洁是以牺牲领域insight为代价的, 让任务完成度上不去, 有价值的往往是长尾的
- 上下文崩溃
	- 当上下文被填充到一定程度时, 模型可能出现大规模遗忘, 召回率降低的问题
	- 极大降低模型的稳定性

### 贡献
>[!核心论点] 
> - 为了实现可扩展的, 持续的自我净化, 不应该把上下文简答的压缩为摘要 
> - 我们应该把context当做是不断演进的剧本, 他们反直觉的提出, LLM更加善于利用冗长的上下文
> - 他们提出了ACE框架, 利用模块化的"生成-反思-整合", 通过结构化的增量更新这个剧本

## Method
### 模块化Agent架构
![[Pasted image 20251028103759.png]]
- 动机
	- 以往的方法将执行-反思-更新都集中在同一个LLM
	- 当被要求压缩上下文时, 容易应多种任务混杂而丢失关键信息
- 工作机制
	- 使用Multi-Agent思路, 独立负责自己的职能
	- Generator:  唯一的任务就是执行(就是原本的LLM), 接受Query与Context Playbook并记录trajectory
	- Reflector: 其任务就是批评, 他分析trajectory以及当前的执行反馈, 并从中抽取见解
	- Curator: 其任务是归档, 他接受Refletor的见解并将其整理为结构化的增量型项目
- 效果
	- 将本来的三大职能进行解耦, 从而有效避免上下文崩溃的情况

### 增量式更新
- 动机
	- 上下文崩溃的根本原因是允许LLM对整个上下文进行重写
	- ACE的推断就是, 不要相信LLM能够无损的重写整个知识库而不出错
- 工作机制
	- 维护一个高度结构化且序列化的Context Book![[Pasted image 20251028104852.png]]
	- Context Book被分为策略, 代码片段, 故障排除等部分, 并且为每个项分配唯一的标识号
	- 这个增量式的更新, 应该是一个轻量级的, 非LLM逻辑的, 并最终确定性的合并到现有Context Book
- 关键实现细节
	- 非LLM合并
		- 知识只能被Add, 而无法删除或者压缩
	- 更新速度比传统的单一LLM速度更快

### 增长与提炼
- 动机
	- 反对使用一个静态的, 简介的系统提示词
	- ACE的目标是演化一个大的, 动态的Context Book, 他主动接受LLM上下文的增长, 这才是积累行业知识的正确途径
- 增长
	- Context Book被设计为支持增量式更新的状态
- 提炼
	- 为了防止无限增长的上下文
	- 避免使用LLM进行总结(这又会陷入上下文崩溃)
	- 而是使用Embedding操作, 计算相似度来保留那些内容上重复的词条
