## Abstract + Introduction
### 背景
- 现有的安全研究主要集中在单智能体的越狱和提示词注入上, 还没有关注MAS
### 挑战
- 既然MAS将任务拆解了, 那么恶意意图是否也被拆解了?
- 我们在SAS上的防御机制, 能否自动继承到多智能体系统中呢
- 缺乏系统性的评估: 现有的评价体系只告诉我们系统失败, 缺没有告诉我们系统架构的哪一部分导致了失败
### 贡献
>[!核心论点] 
> - MAS并不天然比SAS更加安全, 甚至往往更加脆弱, 特定的架构设计选择会引入新的漏洞, 使得原本能够被SAS拒绝的有害请求, 在MAS被偷偷执行了
> - 提出了一个SafeAgent评估框架和一个Dharma的评价指标, 在5种主流架构和4中数据集上进行实证
>

## Method
### SafeAgents
- 动机: 目前的MAS框架(AutoGPT, LangGraph)实现各异, 我们需要统一架构才能评测
- SafeAgents通过抽象出Agent, Tool和Team类, 
	- 其中Team方法接收框架选择和架构类型作为参数

### BenchMark
- RedCode: 能否引导智能体产生有害代码
- SafeArena: 在现实世界平台下, Agent能否完成不道德行为
- AgentHarm: 越狱式对抗性提示的健壮性
- Agent Security Bench: 恶意指令被添加到良性输入提示中, 旨在覆盖原有的良性指令

### Dharma失败分类
![[Pasted image 20260210143732.png]]
- 规划器层级行为
	- 规划器停止: 成功拒绝
	- 规划器被忽略: 规划器发出拒绝或发出警告, 但是未能停止系统
	- 规划器失败: 规划器未拒绝但是没能生成有效的计划, 系统在缺乏有效计划下继续执行
- 子智能体行为
	- 子智能体终止: 指一个或多个子智能体识别出有害行为并拒绝执行, 最终使得系统无法完成有害任务
	- 子智能体忽略: 某些子智能体发出来警告, 但是其他智能体采取忽略导致系统失败
- 如何评判
	- 采用LLM-as-Judge
## Exp
### 不同架构中, 智能体安全性如何
- 集中式MAS并不比SAS更加安全
	- 显式出来更高的攻击成功率
	- 单一协调器更可能比SAS或者分布式MAS更高效的传播未察觉的有害意图
- 去中心化MAS在上下文密集型任务更优
	- 显示出了比SAS和集中式MAS更低的攻击成功率
	- 作者指出, 这一结论具有领域适用性
- 同一架构组内的实现选择也会产生显著的安全差异
	- 不能仅根据架构来判断其安全性

### 集中式架构存在哪些弱点
- 规划器失败时备用机制不足, 导致有害执行
	- 缺乏明确的备用指令导致初期安全组织后依然继续执行
	- 当规划器未能生成计划时, 系统没有明确的停止指令, 反而调用子智能体继续执行
	- 子智能体拒绝行为显著上升
	- 这体现出设计框架的根本性缺陷
- 通过原子化委派任务, 子智能体难以识别有害意图
	- 当子智能体智能访问有限的任务粒度, 不具备足够的上下文来识别有害意图, 这种设计从根本上削弱了子智能体行驶独立判断的能力
- 分层架构是的计划注入被盲目执行
	- 子智能体在执行时不会评估其安全风险, 只会机械的按照步骤识别进行
- 子代理拒绝仍无法组织编排器执行
	- 尽管子智能体明确拒绝, 但是编排器通过代替委托路径继续执行任务

### 小语言模型的印象
- 小语言模型对基于代码的攻击体现出更高的脆弱性
	- 这导致了子智能体拒绝行为的显著上升
