## Abstract + Introduction
### 背景
- Repo-Level的上下文窗口很容易撑爆, 不能直接把整个仓库扔给模型
- 现有的方法通常将代码库视作文档库, 通过检索工具来寻找相关代码片段
- 为了弥补模型能力不足, 研究者往往需要设计大量的辅助工具
- 训练这些Agent通常需要依赖闭源模型的轨迹进行SFT

### 挑战
- 工具也实际执行脱节: 现有的搜索工具是基于文本或高级抽象的, 但这并不符合实际代码执行的逻辑, 比如代码是一行行执行以及存在依赖关系
- 工具越多, Agent的决策空间更大, 出错概率也就越大
- 基于闭源的SFT成本高
### 贡献
>[!核心论点] 
> - RepoNavigator: 只配备了jump(转到符号定义位置)这一个工具的Agent
> - 纯RL训练: 跳过SFT, 直接使用GRPO算法在预训练模型上端到端的RL

## Method
### Jump Tool
- 动机: 作者认为现有工具其本质都是顺序执行与跳转
- 工作机制
	- jump工具直需要Agent提供文件路径和符号, 就能够返回该符号的精确定义代码
	- 机制上模仿了人类使用IDE点击跳转的机制
	- 通过入口点不断jump, Agent访问的代码范围是一执行流为核心的字体, 其信噪比高于全库搜索
- 实现细节
	- 利用Python的静态分析构建AST和符号表

### 端到端 GRPO
- 动机: 之前的观点认为Agent需要先通过SFT启动, 才能够进行RL, 作者认为预训练模型(Qwen2.5)已经有足够的能力, SFT反而会限制模型探索能力
- 工作机制:
	- GRPO
	- 推理->执行(Act/Jump)->观察->奖励
- 奖励函数
	- $$R(Y, Y^*, \tau) = DICE(Y, Y^*) + S(\tau)$$

### 具体执行
- 有模型发出json格式的动作请求, 将对应的文件路径与符号请求发送到解析器中
- 解析器返回结构化的信息给Agent, Agent决定下一步是Reason还是Act
- 总体遵循Reason->Act->Observe循环
