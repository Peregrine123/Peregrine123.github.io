### 长时间运行Agent的困境
- 上下文窗口的限制下, 他们必须以离散的会话方式工作, 而且每个新会话都"对过去毫无记忆"

### 他们的解法
- 创建一个双组件方案
	- 一个初始化智能体: 负责首轮环境搭建
	- 一个编码智能体: 负责在每个对话中增量推进, 并为下一次会话留下清晰产物

### 长运行智能体的问题
- 理论上, 我们借助压缩功能, 我们可以让智能体在持续执行任务中上下文窗口,
- 但是仅靠压缩并不够, 只给一个高层提示(如构建claude.ai的克隆), 难以生成工业级的web应用

### 失败模式
- 智能体一次性做太多事
	- 本质是试图一下子梭哈整个app, 但这常导致模型在实现中途耗尽上下文, 下一个会话接收时只能面对功能只做了一半且没有文档的状态, 导致下一个智能体不得不猜测之前发生了什么, 并花大量时间把基础应用修到可运行
- 部分功能完成后, 后续智能体实例环顾一圈, 看到有进展, 直接宣布项目任务完成

### 拆解问题
- 第一, 我们需要搭建一个初始环境, 为提示中要求的全部功能打好基础
- 第二, 我们应提示每个智能体在朝目标做增量推进的同时, 在会话结束时把环境留在干净状态
	- 干净状态指的是, 代码质量可达到合并主分支的水平, 开发者可以继续开发

### 环境管理
#### 功能清单
- 为解决智能体一次性完成整个应用或者过早宣布项目完成的问题, 我们提示智能体写一个完整的功能需求文件, 把用户初始提示扩展成细致的功能列表
- 他们采用json作为文件格式, 相较于markdown, 模型更不易不恰当的改动json文件
#### 增量推进
- 在初始化环境后, 下一版智能体被要求一次处理一个功能, 这对于解决智能体一次做太多的倾向至关重要
- 即使是增量开发, 模型在代码变更后把环境留在干净状态仍然很关键. 我们的实验发现，最能稳定诱导这种行为的方法，是要求模型把进展以描述性 commit message 提交到 git，并把进展摘要写入进度文件。这使模型可以利用 git 回滚错误改动，并恢复代码库的可工作状态
