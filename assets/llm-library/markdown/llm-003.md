## Abstract + Introduction
### 背景
- LLM在上下文利用方面发展了多种方法:
	- prompt, RAG, 长上下文, 记忆, 工具, agent等

## Why Context Engineering
### 动机
- 超越prompting engineering: 传统的提示词工程已经不足以涵盖涉及, 管理和优化LLM的信息输入范畴
- LLM面对的不在是简单的字符串, 而是动态的, 高度结构化, 多来源的信息流, 需要更加工程化的实现来解决

### 现状局限性
- 纯长上下文方案计算成本大: n方的复杂度无法避免
- 可靠性问题: 纯LLM有幻觉现象
	- 提示词工程的局限: 尽管提示词工程十分重要, 但他缺乏动态性, 系统性

### 优势
- 使用[[In-context learning]]无需更新模型参数, 无需训练即可让模型适应新任务, 时间与人力成本低
- 使用[[CoT]]技术, 让模型具备更加细致的理解, 规划能力
- 利用上下文优化策略: 过滤. 选择, 引导LLM的行为

## Foundation Component

### 上下文检索(Retrieval)与生成
#### 关键构建
- 提示词工程与上下文生成
	- 设计系统提示词
	- zero-shot与few-shot学习
	- CoT分解任务, 自主规划
- 外部知识检索
	- RAG基础: 结合参数化和非参数化手段, 降低幻觉, 提供领域知识
		- FlashRAG, KRAGEN, ComposeRAG
	- 知识图谱集成 KG
	- 高级RAG
		- PARTOR(分层处理)
		- HippoRAG
- 动态上下文集成
