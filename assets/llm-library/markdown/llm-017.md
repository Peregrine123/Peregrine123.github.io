### 三种多智能体优于单智能体的场景
- 当上下文污染导致性能下降时
- 当任务可以并行时
- 不同子任务确实需要不同专业化能力（工具集 / 系统提示词 / 领域知识）

### 为什么应该从单智能体入手
- 良好的单智能体+合适工具, 往往能够达到极强的能力
- 多智能体会引入
	- 更多的故障点
	- 更多不可预期的来源
	- 更多通信和状态传递

#### 典型案例: 规划-执行-评审-迭代
- 现状: 在多智能体在交接处频繁丢失上下文, 最终把token花在协调而非执行上
- 在同等任务下，其token消耗是单智能体的3-10倍
- 这些开销来自：上下文重复、智能体间协调消息、交接摘要的额外成本。

### 何时使用多智能体?
- 只有单智能体明显无法胜任, 或者是证明其带来的额外成本是值得时

#### 上下文保护
- LLM只有有限的上下文窗口, 随着上下文窗口的增长, 相应质量可能会下降

- 场景: 检索历史订单以诊断技术问题的客服代理
	- 单一智能体下, 他要在几千条历史订单信息中反复查找, 光查找过程就占据大部分的上下文, 但是我们只需要最终的结果, 这些过程上下文是无用的
	- 主智能体只需要得到子智能体的50-100token的摘要即可, 子智能体的检索过程对主智能体进行隐藏

#### 并行化
- 并行智能体可以覆盖更大的搜索空间, 适合搜索与研究型任务

- 场景: DeepResearch
	- 主智能体拆分问题维度, 子智能体并行检索, 各自返回压缩结论, 再由主智能体综合

- 注意权衡：并行提升的是 **覆盖度与完备性**，不一定提升端到端速度。多智能体通常仍会消耗单智能体 **3-10 倍 token**，且由于总计算量增加，整体耗时往往并不更短。

#### 专业化
- 某些任务确实需要不同的工具集, 领域提示词和领域知识. 把几十个工具全塞给一个智能体, 常导致工具选错. 按职责创建专用智能体可提升稳定性

- 场景
	- 工具集专业化
		- 当工具集过多时, 模型性能出现显著下降
		- 数量过多(20+)导致选错工具
		- 跨域混淆时
	- 系统提示词专业化
		- 不同的人设和约束经常冲突, 客服需要共情, 写代码需要严苛精确, 合规审查需要刚性规则, 头脑风暴需要发散, 专职智能体+定制系统给提示词通常更稳
	- 领域知识专业化
		- 法律, 医疗等深领域知识会淹没通用智能体, 更可行的方式是: 专业智能体高度承载专有上下文, 按需参与流程

### 如何做任务拆分
- 作者建议按照上下文为中心拆分, 而非问题类型中心拆分
	- 问题类型拆分: 一个写代码, 一个写测试, 一个做评审, 看似分工明确, 实则交接上下文频繁丢失上下文
	- 上下文中心拆分: 谁拥有上下文, 谁也完成相应的测试, 仅当上下文可真实隔离才拆分

- **推荐拆分边界：**
	- 独立研究路径（如亚洲趋势 vs 欧洲趋势）；
	- 有清晰接口的独立模块（前后端按 API 契约并行）；
	- 黑盒验证任务（验证器只跑测试并汇报结果）。

### 高复用模式: 验证子智能体
- 让验证智能体只负责测与验, 不负责做
- 作者也指出, 当编排器足够强时, 他可能直接做出评估, 无需独立验证器, 但在一下场景仍有实际意义
	- 编排器能力较弱
	- 验证本身需要专门工具链
	- 工作流有强制验证关卡

#### 常见失败案例: 过早宣布成功
- 现象: 只跑了一两个案例就宣告成功
- 解决方案:
	- 标准化验证流程
	- 用强制语句约束验证流程
