## Abstract + Introduction
### 背景
- RAG: 主流方法(如SWE-bench的baseline)是非交互式的, 系统检索相关文件, 其一次性喂给LLM, 要求模型生成补丁
- 基础Shell交互: 一些Agent直接使用Linux指令, 如ls, cat, grep来探索代码库
- 普遍共识: 等待基模的上下文性能
### 挑战
- 人机接口的不匹配, 现有系统大多是GUI或者CLI是为人类设计的, 不适合Agent操作
- 上下文是一种稀缺资源: 当Agent使用cat查看一个大文件时, 容易立即占满上下文
- 编辑的困难: 在shell中直接修改代码十分困难, 并且缺乏有效的方法反馈修改错误的原因
### 贡献
>[!核心论点] 
> - 我们需要设计适合LLM Agent的接口, 即ACI, 来让模型发挥出最大的潜力
> - 通过定制化的命令: search_dir, edit和结构化的反馈方案, 将复杂的软件工程转化为模型易于理解的方式
## Method
### ACI的原则与实现
- 搜索
	- grep -r "term" . 可能返回上千行接口, 爆上下文
	- search_dir "term": 限制只返回50行, 如果结果过多, 系统会提示:请细化搜索结构
- 浏览
	- cat file.py 显示全文件: 爆上下文
	- open file.py 仅显示至多100行, 并提供上划和下滑的指令
- 可折叠的上下文
	- 为了避免过长的上下文, 系统会将前几轮的结果折叠为简化的占位符
### 带有Linting的编辑器
- 流程
	- Agent:edit start_line:end_line replace_text
	- 系统在沙箱中模拟
	- 关键步骤: 运行Linter
	- 分支逻辑
		- 0 error 提交更改
		- has error 拒绝更改, 根据系统反馈, 继续修复
