## Abstract + Introduction
### 背景
- 大多数现有的code benchmark都是集中在新代码生成或者修复bug
- 重构基准大多数是原子化的(只设计单一变换), 而且高度依赖python, 对企业级语言(如java)的深度支持较少
### 挑战
- 语义保持: 重构要求在改变程序结构的同时, 不改变外部行为, 这对llm的逻辑一致性提出了极高要求
- 仓库级上下文的缺失: 真实重构往往包含跨文件, 跨类的修改, 仅提供的单个函数片段, LLM无法完成任务
- 噪声干扰: 现实中的Commit往往将重构和新功能混在一起, 如何自动提取纯净的重构案例, 也是挑战
### 贡献
>[!核心论点] 
> - 评测LLM重构能力必须建立在仓库级别, 有真实开发者编写并且经过纯化处理的基准上
> - 作者构建了SWE-Refactor基准, 从18个java项目中自动挖掘1099个重构案例, 并且通过AST静态分析和测试套件的双重验证, 引入了复合重构场景


## Method
### 样本构成
- 目标方法: 重构前的原始方法代码
- **重构类型 (Refactoring Type)：** 明确指定的重构操作
- 仓库与代码结构
- 开发者编写的参考代码
- 构建配置
- 测试覆盖率
### 评估体系
- 功能验证：编译与测试成功
- 语义验证：基于 AST 的重构核查
