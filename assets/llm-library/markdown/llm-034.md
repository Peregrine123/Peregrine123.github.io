### 上下文工程的定义
- 在LLM推理期间, 围绕如何规划并维护最优token集合的一整套策略
- 他区别于提示词工程的是
	- 提示词工程是为了获取最佳结果而编写, 组织LLM指令的方法

### 为什么上下文工程对构建强Agent至关重要
- 随着上下文窗口中的 token 数增多，模型从该上下文中准确回忆信息的能力会下降
- 上下文必须被视作一种边际收益递减的有限资源
- 模型的注意力模式来自训练数据分布，而训练中通常短序列多于长序列。这意味着模型对长上下文依赖的经验更少，也缺乏足够针对性的参数
- 位置编码插值技术, 会让模型对少数token的理解发生退化

### 高效的上下文构成
- 好的上下文工程就意味着：找到尽可能小、但高信号的 token 集合，让目标结果出现的概率最大化
- 系统提示词的编写
	- 应当足够清晰，并使用简单直接的语言，在对智能体而言*合适的抽象高度*呈现信息
	- 一个极端是工程师在提示词里硬编码复杂且脆弱的逻辑来逼出精确的智能体行为，这会导致系统脆弱、维护复杂度随时间上升
	- 另一个极端是只给出模糊的高层指导，既没有给 LLM 提供明确输出信号，也错误地假设了共享上下文
- ![](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2F0442fe138158e84ffce92bed1624dd09f37ac46f-2292x1288.png&w=3840&q=75)
	- 建议把提示词组织成清晰区块（如 `<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` 等），并用 XML 标签或 Markdown 标题来划分，尽管随着模型能力提升，具体格式的重要性可能会下降
- 常见的失败模式之一, 就是工具集过于臃肿, 让LLM要用哪个工具百年的模糊
- 我们不应该穷尽所有的corner case, 硬编码到模型当中
- 提供示例（即 few-shot prompting）是广为人知的最佳实践，我们仍强烈建议这样做


### 上下文检索
- 遵循按需加载的原则, 在运行时通过工具吧数据动态的加载到上下文
- 模型可以编写定向查询、存储结果，并利用 head、tail 等 Bash 命令分析海量数据，而无需把完整数据对象装入上下文

### 面向长时任务的上下文工程
- 对于持续数十分钟到数小时的任务（如大型代码库迁移或全面研究项目），智能体需要专门技术来绕过上下文窗口大小限制。
- 应对方法: 压缩（compaction）、结构化笔记，以及多智能体架构。
#### 压缩
- 压缩是指：当对话接近上下文窗口上限时，对其内容进行总结，并以该摘要重启一个新的上下文窗口
- 压缩通常是上下文工程中提升长期连贯性的第一杠杆。其核心是在高保真前提下提炼上下文窗口内容，使智能体在性能几乎不下降的情况下继续工作
- 以 Claude Code 为例，我们会把消息历史传给模型，让其总结并压缩最关键细节。模型会保留架构决策、未解决 bug、实现细节，同时丢弃冗余工具输出或消息。然后智能体在这份压缩上下文加最近访问的 5 个文件基础上继续工作
- 一个容易下手的冗余内容例子是清除工具调用及其结果——当工具调用已深埋在历史中，为什么智能体还需要再次看到原始结果？最安全、最轻量的压缩方式之一就是清理工具结果

#### 结构化笔记
- 是一种让智能体定期写入笔记并持久化到上下文窗口之外内存的技术。这些笔记会在后续时刻再被拉回上下文窗口。
- 这一策略能以极低开销提供持久记忆

####  Sub-Agent
- 每个子智能体都可能进行大规模探索，消耗上万 token 甚至更多，但只返回浓缩后的工作摘要（通常 1,000–2,000 token）。
