## Abstract + Introduction
### 背景
- 主流的code Agent 在解决仓库级问题时，普遍采用迭代编辑、工具调用和**自主编写测试**进行验证的工作流
- 在SWE-bench榜单排名靠前的Agent几乎都采用了在工作流中进行测试的范式
### 挑战
- Agent编写测试到底显著提升 了任务解决率, 还是仅仅模仿人类开发者的习惯
- 自动生成和执行测试会消耗大量的API调用, 如果测试的反馈价值比较低, 那么他可能成为累赘
- 作者观察到某些顶尖模型(GPT-5.2)在几乎不写新测试的条件下, 仍能够达到与高度依赖测试模型相当的性能

### 贡献
>[!核心论点] 
> - 当前Code Agent编写测试是一种模仿人类的行为, 而非任务成功的必然选项, 在LLM code Generation中, 现有测试实践提供的边际效用十分有限
> 	- 测试主要充当**观测通道**（依赖 Print 语句），而非严密的**验证机制**
> 	- 通过 Prompt 强制增加或减少测试行为，任务成功率并未发生显著波动，但资源消耗却有剧烈变化
> 	- 成功与失败的任务轨迹在测试编写频率和强度上表现出极高的相似性

## Method
### 环境构建
- 仅采用mini-SWE-agent作为轻量级支架, 仅提供基础Bash接口, 不预设测试流程, 以观察模型的原生行为
- 以SWE-bench-verified上进行测试

### 行为指标
- **时机度量 (Timing)**：引入三个归一化指标来刻画测试行为在任务周期中的位
	- **$t_{first}$**：首次编写测试的步数占比
	- **$t_{last}$**：最后一次编写测试的步数占比
	- **$s_{write}$**：测试跨度，即测试行为在整个流程中分布的广度
- **执行强度 (Execution)**：**统计测试执行的总次数**、针对每个测试工件的**平均重跑次数**以及非零返回码导致的**进程级失败率**

### 反馈信号
- 将反馈分为**验证信号 (Assertions)** 和**观测信号 (Value-revealing prints)**
- 仅统计暴露运行时变量或表达式结果的 Print 语句
- 利用规则驱动的 **AST（抽象语法树）分类器**将断言映射到四个层级
	- **C1 Sanity**：仅检查存在性或类型（如 `assert x is not None`）
	- **C2 Property**：检查对象的属性或成员资格（如 `assert hasattr(obj, "attr")`）
	- **C3 Relational**：检查数值范围、约束关系或预期的异常抛出（如 `assert 0 <= score <= 1`）
	- **C4 Exact**：检查精确的结构相等或特定输出值

### 因果干预分析
- 验证测试与成功率之间的因果关系而设计的受控实验
	- **鼓励测试**：针对原生不爱写测试的模型（GPT-5.2, Gemini-3-Pro），在 Prompt 中明确要求编写至少一个可运行的新测试文件
	- **抑制测试 (Discourage)**：针对原生高频写测试的模型（Kimi-K2, DeepSeek-V3.2），删除所有测试相关的引导，并要求其仅通过推理和代码审查来确保鲁棒性
- **转移分析**：不仅看整体成功率，还对比每个任务在干预前后的变化
- 查看Token前后变化

## RQ1：轻量化支架下涌现了哪些测试行为？
### 测试编写频率
- **模型间的极端差异性**：编写测试的行为高度依赖于模型底层训练形成的“风格”
- **成功率的悖论**：尽管测试频率差异巨大，但 GPT-5.2 的解决率 (71.8%) 竟与排名第一、高频写测试（83%）的 Claude-Opus-4.5 (74.4%) 处于同一量级 。这初步证明了**测试编写并非成功的充要条件**
- 在同一模型内，未解决任务的平均测试数量往往多于已解决任务，这反映了 Agent 在遇到难题时更多的**试错（Trial-and-error）行为**

### 测试编写时机
- 作者使用了归一化的步数占比 $t_{first}$（首次编写位置）、$t_{last}$（最后编写位置）和 $s_{write}$（编写跨度）来刻画测试在任务生命周期中的分布
- **普遍的“晚期特征”**：无论模型如何，测试编写通常结束得很晚（平均 $t_{last} \approx 0.80$），意味着测试往往贯穿到任务的最后阶段
### 测试执行强度与进程结果
- **执行负荷**：对于编写了测试的任务，未解决的任务轨迹在测试执行总次数
- **环境摩擦力**：测试执行时产生非零返回码的比例主要由模型能力决定，与任务最终是否解决无关

## RQ2 关注测试代码的实质内容
### 任务级反馈信号量：打印（Print）统治断言（Assert）
- **观测为主导**：在所有研究的模型中，**打印语句的数量始终显著超过断言**
- 未解决的任务通常含有更多的信号，这主要是由额外的打印语句驱动的
- **核心结论**：Agent 编写测试的主要目的不是为了自动化验证补丁是否正确，而是将其作为一种**探测接口**，通过打印值来观察代码运行状况。

### 断言分类：简单的精确检查 vs 缺失的复杂逻辑

| **类别**            | **定义**   | **典型示例**                      | **占比特征**               |
| ----------------- | -------- | ----------------------------- | ---------------------- |
| **C1 Sanity**     | 健全性检查    | `assert x is not None`        | 约 15% - 20%            |
| **C2 Property**   | 属性/有效性检查 | `assert hasattr(obj, "attr")` | 约 33% - 41%            |
| **C3 Relational** | 关系/范围检查  | `assert 0 <= score <= 1`      | **极稀少 (2.9% - 7.9%)**  |
| **C4 Exact**      | 精确值检查    | `assert output == expected`   | **主导地位 (约 41% - 43%)** |
### RQ2 总结
**Agent 的测试能力目前还处于“原始观测”阶段**。 它们大量依赖 Print 来排查错误，而编写的断言大多是简单的相等判断，缺乏对软件行为深度逻辑的约束

## RQ3：Agent 编写的测试真的影响任务解决吗？
- 作者设置两组实验, 让喜欢写测试的抑制其测试行为, 让不喜欢写测试的鼓励其测试行为

### 结局的惊人稳定性
- **结局高度稳定**：平均 **83.2%** 的任务在干预前后保持了同样的解决结果
- **边际效用极低**：
	- **GPT-5.2** 强制写测试后，净成功数变化为 **0** 。
	- **DeepSeek-V3.2** 强制不写测试后，在 376 个原本写测试的任务中，仅减少了 **20** 个成功案例

### 效率与成本的剧烈波动
- 测试行为主要改变的是“资源足迹”，而非“成功概率”
	- **“昂贵的习惯”**：对于 GPT-5.2，增加测试只会显著拉高 Token 消耗，却换不回任何额外的解决数
	- **“高性价比的削减”**：对于 Kimi-K2，不写测试虽然损失了 2.6% 的成功率，但**节省了近一半的输入 Token** 和超过三分之一的 API 调用

### 总结
- 写测试更多是一种“仪式感”而非“生产力”
- **对低频模型**：强迫它们写测试是浪费钱，因为它们并没有学会如何利用测试反馈来改进代码 。
- **对高频模型**：它们写测试的行为存在严重的**边际效用递减**。削减这些测试行为可以极大降低工业化部署的成本，而性能损失却微乎其微 。
