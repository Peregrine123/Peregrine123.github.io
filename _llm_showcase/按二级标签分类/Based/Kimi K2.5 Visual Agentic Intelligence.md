## Abstract + Introduction
### 背景
- VLLM方面
	- 大多数多模态模型, 采用的是LLAVA范式, 即外挂一个ViT视觉编码器, 通过微调来对齐
- Agent方面
	- 主流的Agent框架, 主要依赖串行执行策略
### 挑战
- 在训练中映入视觉数据往往会损害模型的文本推理能力
- 随着任务复杂度的提升, 串行Agent的推理时间线性甚至指数增长, 极易撑爆上下文窗口
- 视觉Agent数据缺乏: 高质量的视觉-动作-推理数据机器稀缺, 导致多模态Agent的冷启动十分困难
### 贡献
>[!核心论点] 
> - 真正的多模态能力不应该是简单的模块凭借, 而应该通过早期联合训练实现模态间的融合; 同时, 复杂任务的解决不应该依赖单体Agent的线性堆叠, 而应该通过Agent Swarm(集群)实现现代计算集群的并行并发
> 	- 联合优化: 提出早期融合的预训练策略和Zero-Vision SFT零视觉微调, 利用文本能力唤醒视觉推理
> 	- Agent Swarm: 基于RL训练的指挥官(Orchestrator)架构, 动态创建和调度子Agent并行工作

## Method
### 原生多模态联合
- 动机: 传统的后融合策略(在预训练后期加入大量视觉数据)会导致文本能力的震荡, 作者观察到先跌后涨的现象, 此外, 缺乏高智力的视觉文本推理数据使得视觉与文本推理泛化变得困难
- 工作机制: 
	- 视觉数据早期融合: 在预训练早期就混合视觉与文本数据
	- 视觉数据低比例: 作者发现, 保持10~20%的低比例, 能够让模型的视觉能力得到较大提升
	- Zero-Vision SFT: 作者使用纯文本的SFT数据来训练模型的视觉Agent能力
		- 所有的视觉操作(裁切, 旋转, 目标检测)都被抽象为python代码调用, 模型在文本中学会图像位置, 并学习如何写代码去处理
	-  

### 基于结果的视觉RL
- 将任务分为三个领域
	- 视觉定位与计数
	- 图表理解与文档理解: 对结构化视觉信息进行解释及文本提取
	- 视觉关键的STEM问题: 经过筛选, 必须依赖视觉输入的数学和科学问题
- 通过这些轨迹进行拒绝采样微调, 可构建一个自我优化的数据管道


### Agent Swarm
![[Pasted image 20260204151839.png]]
- 动机: 不依赖启发式的并行化规则, 而是通过动态任务拆解, 子智能体实例化与并行子任务调度来启动智能体集群. 关键在于, 并行性并非设置的必然优势, 关于是否并行, 何时并行以及如何并行的决定, 均通过环境反馈以及RL学习而非显示
- 工作机制
	- Orchestrator: 一个专门经过RL训练大脑, 他负责任务分解和子智能体创建
	- Worker: 被动实例化, 参数冻结模型副本, 他们拥有独立的, 短暂的上下文, 只负责处理切分后的小任务
	- 上下文分片: 每个子智能体制维护自己的局部记忆, 只有最终结果汇总给Orchesrator
- 并行智能体强化学习(PARL)
	- 作者设计了一个特殊的RL奖励, 为了避免Multi-Agent中的信用分配的问题(也就是不知道哪个子智能体失败了), 作者冻结了子智能体, 只训练Orchestrator
	- 基于结果的奖励本质是稀疏且含噪声的, 在多智能体的设定中, 一个正确的答案并不意味着所有子智能体的执行都正确, 而一个错误的答案也不意味着所有智能体出错, 通过冻结子智能体并将其视为环境观测而非可微分的决策点, 将高层能力与底层任务能力解耦
- **核心公式解析：**$$r_{PARL}(x,y) = \lambda_{1}\cdot r_{parallel} + \lambda_{2}\cdot r_{finish} + r_{perf}(x,y)$$
	- $r_{perf}(x,y)$: **结果奖励**。最终任务做得好不好？这是基础。
	- $r_{parallel}$: **并行奖励**。这是关键创新。如果系统发现多个子任务可以并行执行，且指挥官确实选择了并行，就给予正反馈。这防止模型“偷懒”退化回单线程模式。
	- $r_{finish}$: **完成率奖励**。防止模型为了骗取并行奖励而疯狂创建无效的子智能体。这要求子任务必须是可执行且能完成的 。
- 资源消耗约束
	- 为了衡量计算时间成本, 通过类比计算图中的关键路径来衡量关键步骤
	- 通过观测关键步骤而非总步骤的做法, 有效促进了任务拆解
