## Abstract + Introduction
### 背景
- 随着GPT-4的出现, 从单纯的代码补全转向自主解决问题的Agent, 他们用来设计为模拟人类开发者, 使用shell命令
- 视代码为文本: 将许多的代码库视为文件的集合, 主要依赖文本匹配和自然语言理解来定位
### 挑战
- 检索的非结构化: 单纯使用grep或文件读取往往效率低下, 并且容易内容过多爆上下文
- 试错成本高: 现有的Agents, 通常基于观察, 思考, 行动范式, 执行大量的shell命令, 造成了极高的等待时间以及token成本
- 缺乏传统软件工程技术的结合
### 贡献
>[!核心论点] 
> - 我们不能把代码单纯看做是文本集合, 而是应该利用程序表示(如AST)和程序结构进行进行导航
> - 构建基于抽象语法树(AST)的分层上下文检索机制, 并结合基于频谱的故障定位, 代替了传统shell
> - LLM决策+符号化分析工具的范式

## Method
### 分层上下文检索
- 动机
	- 直接搜索会返回上千条结果, 效率低下
	- 模仿人类, 先看大纲, 再看细节
- 工作流程
	- LLM分析issue, 提取关键类名所谓线索
	- 广度优先搜索: 根据类名查找类内的代码, 不返回所有代码, 只返回类内的属性和方法
	- 深度聚焦: 根据上一步获得的结果, 针对性的解析, 精准切取代码返回给LLM
- 他实现了一组专用API
	- search_class(cls)
	- search_method_in_class(m, cls)

### 分析检索增强
- 动机
	- 用户描述的ISSUE往往不够精确, 通过现有的测试套件来透视代码的执行路径
- 工作机制
	- 前提: 存在测试案例
	- SBFL运算: 计算每个方法的可疑度分数
	- 将top-k个结果返回给llm, 作为额外线索
	- 融合推理: 根据issue和SBFL的结果, 决定下一步搜索到哪里
