## Abstract + Introduction
### 背景
- Copolit范式: 补全为主
- 早期的Agent尝试: 借助ReAct范式
- Claude code, Cursor范式
### 挑战
- 代码复杂度的上升: 要求agent不在只是补全函数, 而是写出一个完整项目
- 上下文过载
	- 如果生成整个项目, 需要输入充分的文档, 以及已有的代码
	- 直接放入会撑爆上下文窗口, 信噪比过低

### 贡献
>[!核心论点] 
> - 仓库级代码生成不应该是做简单的文本生成, 应该看作是信息流管理问题
> - 提出完整的信息流操作流程
> 	- 源压缩
> 	- 结构化索引
> 	- 条件知识注入: RAG
> 	- 闭环纠错: 验证反馈
> 
>
## Method
### 基于CodeMem的状态化生成
- 动机: 在生成第N个文件时, Agent需要读取N-1个文件, 这样会导致上下文爆炸
- 工作机制
	- 维护一个动态化, 结构化的记忆库
	- 压缩与索引: 每生成一个文件, 就会启动总结Agent, 将代码有损压缩为一个记忆库的item
	- item内容: 核心意图, 公共接口, 依赖: 相当于一个api文档
	- 动态检索: 生成新文件时, Agent直接从记忆库中检索相关摘要

### 源压缩与蓝图蒸馏
- 动机: 由于任务是paper2code, 原始的pdf是多模态, 非结构化的, 信噪比低
- 工作机制
	- 在编写第一行代码前, 进行分层内容分割和多agent分析
		- Concept Agent负责宏观架构, Algorithm Agent负责公式与超参数
	- 合成蓝图: 升成一个标准化的实现计划, 这个部分由专门的Planing Agent负责
	- 代码生成阶段
		- CodeMem
			- 上下文构建: 上下文并不是基于源代码, 而是从动态选取的记忆子集构建而成, agent从代码库中检索哪些与当前代码文件相关, 然后只获取他们的摘要记录
			- 然后进行代码生成
			- 更新记忆库: 由专门的摘要agent负责
		- CodeRAG
			- 相关性过滤: 从参考论文的代码库, 或者是公开仓库中检索相关信息
			- 代码理解: 生成结构化索引, 并且生成这些信息的摘要
			- 关系映射: 从索引的代码和我们目标生成的代码建立索引
	- 验证阶段
		- 静态检查
			- 无需代码执行即可解决的问题, 由分析智能体和修改智能体完成
		- 沙盒执行与功能修正
			- 环境验证与配置: 沙箱agent根据蓝图中的描述进行环境配置, 如果遇到问题就修改相关配置的文件
			- 迭代执行与生成: 找到主入口后, 利用自动生成的测试代码检验算法与功能, 并生成错误的轨迹信息
			- 这将启动一个迭代优化循环: 持续迭代知道达到最大迭代次数或成功

### 输入输出元素
- 输入
	- 有多模态信息形成的块, 构成的集合
- 输出
	- 文件树
	- 代码文件
	- 依赖

### 整体流程架构
- 实现计划撰写
	- 将多模态输入信息, 转化为结构化, 高信噪比的实现计划
- 代码生成阶段
	- CodeMem: 对源代码进行结构化索引, 维持跨文件一致性
	- CodeRag: 执行条件知识注入
- 自动化验证
	- 将执行反馈作为验证信号, 通过闭环纠错机制修复bug
