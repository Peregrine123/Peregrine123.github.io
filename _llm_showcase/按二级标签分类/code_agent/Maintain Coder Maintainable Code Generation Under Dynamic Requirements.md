## Abstract + Introduction
### 背景

### 挑战
- 现有系统忽略了SE中最重要的维度--可维护性
- 现有的多智能体方法(AgentCoder, MapCoder) 仅优化任务完成度, 往往生成结构脆弱, 耦合度高且难以维护的代码
- 缺乏量化代码维护成本的动态基准测试
### 贡献
>[!核心论点] 
> - 代码生成应该通过架构设计来应对不可避免的需求变更
> 	- 提出了MaintainCoder, 结合瀑布模型, 经典设计模式和多智能体协作, 系统性的增强代码内聚性并降低耦合度
> 	- 建立了新基准MainTainBench定义了动态维护性

## Method
### Maintain Bench
- 扩展了 HumanEval、MBPP、APPS、CodeContests 和 xCodeEval 五个经典基准
- 四种需求变更模式 ($\Pi_B$)
	- **功能扩展 (Text)：** 引入额外的现实需求，要求调用原函数作为子组件
	- **接口修改 (int)：** 修改输入参数、返回类型等协议，考察代码的鲁棒性
	- **数据结构变换 (dst)：** 明确要求采用不同的底层数据结构（如从列表改为树），模拟性能优化场景
	- **错误处理增强 (err)：** 引入具体的异常处理机制，反映系统的可靠性

### Maintain Coder
![[Pasted image 20260215101414.png]]
- 分成Code Framework以及Code Generation两个模块
#### Code Framework
- **需求分析智能体：** 运用思维链分解问题，提取关键目标，并提出高层逻辑方案，避免过早陷入细节 。
- **设计模式选择智能体 ：** 充当软件架构师，根据模块挑战选择最合适的设计模式。
- **框架设计智能体：** 构建初步的类结构，明确类之间的继承、组合与依赖关系，严格遵循单一职责原则 。
- **框架评估智能体 ：** 评审设计的清晰度、扩展性和性能，识别过度碎片化等问题，并反馈给设计智能体进行迭代优化

#### Code Generation
- **代码生成智能体：** 将详细框架转换为完整的 Python 代码，强制要求遵守 PEP 8 规范，并添加针对非显然逻辑的 purposeful 注释 。
- **代码优化智能体：** 执行代码并根据反馈进行迭代调试 。它会通过 CoT 诊断根因，持续重测直至代码通过所有验证

## Exp
### 评估指标
- **静态指标：** 维护性指数 (MI) 和 圈复杂度 (CC)
- **动态指标（核心）：** 修改后的功能正确性 $Pass@5$、抽象语法树相似度 $AST_{sim}$ 和 代码变更量 $Code_{diff}$
