## Abstract + Introduction
### 背景
- GRPO在当时占据统治地位
- 结果导向的公式: 现有的RLVR方法主要依赖结果验证
![[Pasted image 20260121102228.png]]
- Clip: 是否对称裁切更新幅度
- Aggr: 聚合方式: 
	- **smtm (Sequence-Mean-Token-Mean):** 先在序列内求平均，再在组内求平均。这是最标准的做法，它对长短序列一视同仁 。
	- **smts (Sequence-Mean-Token-Sum):** 先把序列内所有 Token 的 loss 加和，再求平均。**Dr.GRPO** 使用此方法，这意味着长序列权重更大 。
	- **tm (Token-Mean):** 把所有 Token 混在一起求平均，忽略序列结构。**DAPO** 使用此方法
### 挑战
- 对于长轨迹任务的奖励系数问题: 对于像Appworld的多步任务, 但凡过程失败, 就导致整个任务失败
- 过程信息的浪费: 代码执行器在过程中会产生丰富的信息, 但是结果验证就无法充分利用他们了
### 贡献
>[!核心论点] 
> - 主张将过程级反馈引入到优势函数中, 实现step-level的奖励分配
> - 提出了GVPO, 融合结果验证与每一步的过程验证
> - 在交互式Agent场景中, 关注执行过程比仅关注结果更有效

## Method
![[Pasted image 20260121105255.png]]
- 动机: GRPO中, 对于同一条轨迹 $\tau_i$，其所有时间步 $t$ 共享同一个优势值 $\hat{A}_{i,t}$
- 工作机制: 
	- 先看最后答案，定一个基准分 $\hat{A}_{i,t}$
	- 再逐行看推导过程。如果某一步出现了显式的计算错误（$B_{i,t}$，如代码报错），就在那一行的基准分上追加惩罚。
	- 最终得分 $A_{i,t}$ 是两者结合的结果。
- 修正了优势函数
	- $$A_{i,t} = f(\hat{A}_{i,t}, B_{i,t})$$
	- 其中 $\hat{A}_{i,t}$ 是基于结果的组相对优势（未标准化的）。 对于过程修正项 $B_{i,t}$，作者根据 Token 是否属于失败步骤（$\mathcal{I}_{i}^{fail}$）进行了精细的定义。
- 惩罚设置
	- 当某一步骤执行失败（$t \in \mathcal{I}_{i}^{fail}$）且当前轨迹的总体结果也是负面（$\hat{A}_{i,t} < 0$）时：$$A_{i,t} = (1+b) \cdot \hat{A}_{i,t}$$
		- 实际上, 加大了惩罚的力度
