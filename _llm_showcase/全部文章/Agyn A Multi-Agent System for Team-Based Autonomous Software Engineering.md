## Abstract + Introduction
### 背景
- **预定义流水线：** 部分多代理系统（如 TRAE, Prometheus）采用固定的执行阶段，缺乏应对复杂软件开发任务所需的动态协作能力
### 挑战
- **职责过载与上下文管理：** 单代理模式在处理长序列任务时，由于职责杂糅（如规划与精确代码编辑的矛盾），极易导致模型推理性能下降或计算资源浪费
- **生产环境适配性：** 如何在不针对基准测试调优的前提下，构建一个能融入日常工程工作流、具备自我纠错和审查机制的自动化团队

### 贡献
>[!核心论点] 
> - 作者认为将软件工程建模为由**经理协调、多角色专业化分工**的组织过程，能够有效提升自主系统在解决真实复杂问题时的可靠性与效率
> 	- 构建了包含 **Manager、Researcher、Engineer、Reviewer** 的多代理团队，通过**隔离的沙盒环境**、**GitHub 原生工作流**以及**基于角色的模型配置**

## Method
### 角色分类
- Manager
	- **运作逻辑**：不直接执行编码，而是应用高层开发方法论
	- **核心职责**：负责整体协调、任务状态维护及决定后续调用哪个代理
- Researcher
	- **核心职责**：探索代码库、识别问题根因，并产出结构化的任务规格说明书
- Engineer
	- **核心职责**：负责具体实现，包括修改代码、运行测试及在隔离环境中迭代方案
- Reviewer
	- **核心职责**：通过 GitHub 的 PR 机制检查代码差异（Diff），留下行内评论并给出“批准”或“请求修改”的显式信号

### 隔离环境
- **Nix 软件包管理**：代理不仅拥有 Shell 访问权限，还配备了 **Nix** 运行环境 。这允许代理根据项目需求动态安装依赖或调整环境版本（如降级 Python），解决了预设环境过于僵化的问题
- **大容量输出处理**：当 Shell 命令输出超过 **50,000 token** 时，系统会自动将其重定向至临时文件并返回引用，防止冗余信息塞满上下文

### 通信机制
- **经理中心化通信**：代理之间不直接对话，所有信息流必须通过 Manager 调用的 `manage` 工具进行中转 。这种设计简化了控制流并增强了任务的可追溯性
- GitHub 原生工具链
	- **身份隔离**：为每个角色创建独立的 GitHub 账号
	- **命令行工具**：弃用元数据冗余的 API 接入，改用 `gh` 命令行工具以获取更紧凑、适合模型阅读的输出
	- **自定义扩展**：开发了 `gh-pr-review` 工具，专门支持代理阅读和撰写 PR 行内评论，从而参与迭代评审循环

### 自动化与长程执行策略
- **强制自治指令**：在 Prompt 中严禁代理请求人类批准或确认，必须自主完成任务
- **控制流分离**：
	- 使用专用的 `finish` 工具信号标志任务终结 。
	- 引入 `send_message` 工具允许代理发送中间状态，而不触发对话终止
