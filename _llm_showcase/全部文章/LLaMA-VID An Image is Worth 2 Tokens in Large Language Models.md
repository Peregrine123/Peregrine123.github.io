## Abstract + Introduction
### 背景
- 主流的多模态模型为图像任务而设计(BLIP2, LLAVA), 他们需要高昂的tokens消耗来表示一张图片
- 静态编码: 主流模型倾向于把图像编码为固定的特征序列, 无论用户问什么, 序列都是固定的
### 挑战
- 在视频领域的token爆炸: 如果按照图像模型一帧256tokens, 那么一段包含10k帧的视频将有高达3M的视觉tokens, 这是LLM上下文无法承受的
- 信息密度与效率的权衡: 为了处理长视频, 常用时序降采样或者特征平均, 这样虽然减少了token消耗, 但是严重破坏了长时序视频内细粒度特征. 
### 贡献
>[!核心论点] 
> - 视频中的每一帧可以通过两个不同性质的token: Context token和 Content token, 来进行高效的表达, 从而使得LLM能够处理小时级别的论文
> - 作者设计的双token策略, 其中Context Token 利用作者指令提取动态信息, Content Token则保留视频本身纹理信息
## Method
### 双token机制
#### 动机
- 当模型不知道用户询问的目的, 就会尽可能的记下所有的视觉token, 造成冗余
- 我们如果能够知道用户询问的token, 就能够减少大量无关token
#### 工作机制
对于视频中第t帧
- 输入: 图像帧以及用户文本指令
- 视觉特征提取, 并进行文本查询
- 让视觉和文本交互, 生成1个Context token
- 为了防止指令没有覆盖到的信息, 我们在利用视频特征进行池化, 为了极值效率, 我们可以保留仅1个token
- 融合: 最终的每帧表示为 \[Context token, Context token\]
#### 数学机制
- 刚才视觉和文本的交互, 论文使用了一个非常清晰的注意力公式来定义 Context Token 的生成。这是理解其“指令感知”特性的关键：
$$E_{t}=Mean(Softmax(Q_{t}\times X_{t}^{\top})\times X_{t})$$
- 这里 $Q_t$ 是文本查询，$X_t$ 是视觉特征。
