## Abstract + Introduction
### 背景
- 传统LLM代码生成: 指没有环境交互, 只依赖用户输入prompt进行反馈, 缺乏感知, 规划, 自主行动的能力
- LLM Coding Agent: 已LLM为核心大脑, 具备自主性, 任务扩展和工程实用性增强的特点
	- 模拟人类程序员的完整工作流: 需求分析->规划->编码->测试->修复
	- 从单纯的代码撰写者转变为任务定义者以及结果审查者

## 关键技术体系
### 单智能体
- 规划和推理(Planing and Reasoning)
	- self-planning: 在写代码之前进行计划制定
	- 多路径搜索: 借助MCTS或者思维树(ToT)来探索不同方案
- 工具集成与检索增强(Tool Use & RAG)
	- IDE, Linter, API-search
	- RAG: 针对代码库的检索, 解决上下文的限制和幻觉问题
- 反思与自我改进(Reflection & Self-improvement)
	- 模型生成代码后, 自我审查或者通过执行反馈进行迭代修复

### 多智能体
> 多智能体通过角色分工解决更加复杂的问题
- 工作流模式
	- 流水线: 如瀑布模型, Chat-Dev(设计-编码-测试)
	- 层级式: 上层规划, 下层执行
	- 自协商与自我进化: Agent之间互相讨论甚至动态调整团队结构
- 上下文管理
	- 使用黑板模型, 共享全局信息
	- 模拟短期与长期记忆
- 协作优化
	- 指Multi-Agent通过相互评估和辩论来提升代码质量, 减少错误传播

## 细致探讨
### 规划与推论架构
- 线性规划
	- 自我规划模式: 根据需求给出一系列的步骤, 并在最终进行代码实现, 问题在于灵活性差, 如果中间规划不足就会导致任务事变
	- CodeChain链式修正: CodeChain引入模块化自修复机制, 他通过自我修订来执行
- 树与图的搜索
	- 蒙特卡洛树搜索
		- 采样: 在每一轮生成中, 不止生成一个结果, 而是进行多次采样构建决策树
		- 评估: 结合代码的执行反馈, 对每个候选分支进行打分
		- 选择: 选择最优路径进行扩展
		- 作用: 实现从单路径规划转向多路径并行推理
	- 代码树
		- 将过程组织为统一的树结构, 显示建模了"策略搜索", "解生成"以及"迭代优化"三个阶段, 利用启发式引导剪枝来优化
	- 自适应搜索树
		- 根据代码的执行反馈, 动态的在每个决策节点分支出新的决策节点
- 显式搜索与非序列化环境
	- PlanSearch: 将本身做计划的过程当做搜索的过程, 生成多组候选计划并在推理阶段并行评估, 从而创造更大的解空间
	- 引导搜索: 训练一个动作价值评估器(有点像RL中的$V(s)$), 预测下一步的潜在结果来评估候选动作, 并且选择最有可能性的轨迹
