## Abstract + Introduction
### 背景
- 单智能体: AutoGPT尝试递归ReAct循环, 来解决复杂问题
- 多智能体尝试: CAMEL, AgentVerse, 模拟多智能体相互对话进行交互
### 挑战
- 错误累计: 多智能体自由交流时, 一个幻觉信息容易被传导放大,
- 协作无序: 容易没有明显职责界限, 导致无限对话而无法推进任务
- 长逻辑链路崩溃
### 贡献
>[!核心论点] 
> - 引入人类软件工程范例, 让流程标准化
> - MetaGPT通过元编程, 将SOPs编码到多智能体框架里面, 强制智能体输出标准的结构化输出(PRD文档, API接口设计图)

## Method
### 基于SOPs的编程
 - 动机
	 - 如果不借助PRD(需求文档)和设计图就开始写代码, 容易缺乏顶层设计, 变成纯聊天
 - 基于瀑布式软件开发模型, 
 - 流程
	 - **Product Manager (PM):** 接收一句话需求 $\rightarrow$ 输出 **PRD（包含用户故事、竞品分析）**
	- **Architect:** 接收 PRD $\rightarrow$ 输出 **系统设计图（UML、API 接口）**
	- **Project Manager:** 接收设计图 $\rightarrow$ 输出 **任务列表与依赖关系**
	- **Engineer:** 接收具体任务 $\rightarrow$ 输出 **代码**。
	- **QA Engineer:** 接收代码 $\rightarrow$ 输出 **测试用例**
### 基于角色的共享记忆池
- Meta借助发布--订阅机制, 而不是对话机制
- 环境: 维护一个全局的记忆池
- 角色订阅: 每个角色根据自己定位关注特定消息类型
- 记忆检索: 智能体能够按照当前任务去检索相关的历史记录
