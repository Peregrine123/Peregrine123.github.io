## Abstract + Introduction
### 背景
- 手工设计架构: 现有的高性能Agent System, 如ReAct, CoT, 都是人类工程师手动编写逻辑的
- 有限的元优化: 虽然出现了一些自动优化系统, 但局限于优化自然语言的Prompt, 而不是Agent的代码逻辑
- 分离的优化架构: 采用Meta-Agent来优化另一个Target-Agent, 但范围局限于某个函数, 而非整个Repo
### 挑战
- 探索空间受限: 人类只能探索解空间的一小部分, 如果我们只优化prompt, 二不允许agent修改底层的Python代码, 能力收到限制
- 缺乏真正的自我修正: Agent本身即是执行者, 又是开发者, 如何让Agent在安全, 有效地重写自己的底层代码, 是一个挑战
### 贡献
>[!核心论点] 
> - 让ai具备修改自身底层Python源代码的能力, 从而实现性能提升
> - 他们通过构建自提升的coding agent, 消除了外部agent的介入监督
> - 通过基准测试->归档分析->代码元改进的迭代循环

## Method
### 完全自提升的迭代循环
- 动机: 过往的方法将优化者和被优化者区分开, 并且限制在Prompt的修改上, 作者认为要让Agent直接修改自身的全量python代码库, 打破界限
- 工作机制
	- $A_0$从一个具备基本功能(grep, rw, bash)的基础agent开始
	- 评估: Agent在一组基准任务上执行, 生成性能报告
	- 元改进: 系统从历史存档中选出表现最好的Agent作为元代理
		- 这个Agent被赋予查看存档, 分析任务是否成功, 修改代码库以提升下一次的表现
		- 他拥有归档分析工具, 可以查看历代版本的统计数据
	- 生成 ($A_{i+1}$), 修改后的代码成为下一代 Agent，进入下一轮循环

### Agent的评估优化框架
- 作者定义了一个显式的效用函数来指导进化方向, 效用 $U$ 定义如下：
$$U = w_{score}p_{score} + w_{cost}(1 - \min(1, p_{cost}/\$10)) + w_{time}(1 - \min(1, p_{time}/300s))$$
- $p_{score} \in [0,1]$：归一化的基准测试分数。
- $p_{cost}$：解决问题的美元成本（上限设为 $10）。
- $p_{time}$：墙钟时间（Wall-clock time，上限设为 300秒）。
- 权重配置：$w_{score}=0.5$, $w_{cost}=0.25$, $w_{time}=0.25$。

### 子智能体协调机制
- 在初始化得到基础智能体后, 主智能体会将任务分配给子智能体
- 这是为了更好的管理上下文
- 他们设置了代码智能体, 问题求解智能体以及推理智能体
- 他们还设置了异步监督期, 负责定期调起, 监督是否出现异常状况

### 提示词设计-多智能体
- 主智能体: 只负责路由, 委托, 编排, 不完成任何实质性的任务
- 子智能体: 任务原子化
	- 接受任务->执行->返回结果->销毁上下文
- 使用xml而非json: 在 JSON 中，字符串必须转义（例如 `\n`, `"` 等）。这对于 LLM 来说是一个巨大的负担，经常导致格式错误，或者为了转义消耗大量 Token。

## 启示
- 让Agent不在僵化的通过prompt来走流程, 而是能够自己发明工具, 并且迭代循环筛选出最优的Agent
- 随着ai能力的提升, 应该进一步的开放对Coding Agent的自由度
