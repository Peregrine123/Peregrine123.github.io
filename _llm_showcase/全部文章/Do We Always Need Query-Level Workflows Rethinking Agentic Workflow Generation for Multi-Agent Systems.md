## Abstract + Introduction
### 背景
- 组织MAS有两种工作流生成范式
	- Task-Level: 针对整个数据集(如GSM8K)来搜索出通用最优的工作流, 代表作有Aflow和GPTSwarm, 他们通过搜索算法在验证集上不断迭代
	- Query-Level: 针对每一个具体的输入, 动态生成一个定制化的工作流, 代表有MAS-GPT和ScoreFlow, 这种假设是, 不同的问题需要不同的解决路径
### 挑战
- Task-Level的评估昂贵: 搜索最优工作流时, 每一个候选工作流都要在整个验证集上跑一遍, Tokens消耗高
- Query-Level的推理昂贵: 虽然理论上更加灵活, 但是对每个问题都重新生成工作流会带来巨大的推理延迟和成本
- 这两种工作流方式真的是正确的方向吗
### 贡献
>[!核心论点] 
> - 作者认为, 少数几个高质量的Task-Level工作流组合(top-K)就足以覆盖绝大部分的Query需求
> - 提出了Scale低成本估计框架, 用模型子预测+少量样本校准代替全量执行评估

## Method
### MAS工作流的定义
- 任务 $T$ 包含一系列查询$q \in \mathcal{D}$。一个工作流 $W$ 的作用是指挥基于大语言模型的多智能体系统来生成答案 $y$
- 工作流空间 $\mathcal{W}$ 被形式化地定义为：$$\mathcal{W} = \{(P_1, ..., P_n, \Phi, O_{\theta_1}, ..., O_{\theta_n})\}$$
	- $n$ 是智能体的数量。
	- $P$ 是提示词（Prompt）空间
	- $\Phi$ 是 **信息控制流**
	- $O$ 是一组预定义的、参数化的 LLM 智能体(Review, Ensemble, Self-Correction)
- 度量: 工作流 $W$ 在查询 $q$ 上的表现通过任务特定的评估器 $s(W(q), q)$ 来衡量（例如精确匹配度），得分范围在 $[0, 1]$ 之间

### Query-Level Workflow是否总是重要的
- Top-5在查询覆盖率上获得明显的提升, 证明可以通过少量候选任务来实现Query的性能

### Scale框架
- 预热阶段
	- 先在验证集上全量跑M轮, 这个M是远小于本来需要跑的轮数
	- 这个阶段是为了描绘难度分布
- 评估阶段
	- 自预测 (Self Prediction, $S^{pred}$)
		- 将新生成的工作流 $W_{t+1}$ 代码输入给 LLM 优化器
		- 使用专门设计的 **评估提示词（Evaluation Prompt, $P^{eval}$）**，要求 LLM 预测该工作流在全量数据集上的准确率
	- 小样本执行校准 (Few-shot Execution Calibration, $S^{few}$)
		- **采样策略：** 不随机抽取，而是基于预热阶段计算的“难度分布”进行**分层采样**
		- 将所有验证集问题按难度分成 $K$ 个桶
		- 根据桶的分布抽取极少量样本 $\mathcal{D}_{few}$
		- 仅在这个极小的子集上运行工作流，得到真实执行分数 $S^{few}$
		- 最终的得分是“自预测分数”和“小样本分数”的加权融合：$$\hat{S}_{t+1} = (1 - \alpha_{t+1}) S_{t+1}^{pred} + \alpha_{t+1} S_{t+1}^{few}$$
