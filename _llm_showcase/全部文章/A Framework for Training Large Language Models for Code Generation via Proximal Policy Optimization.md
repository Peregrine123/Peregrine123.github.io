## Abstract + Introduction
### 背景
- 学界普遍使用Actor-Critic架构, 使用PPO算法来提升模型English
- 训练范式: 大多采用数据并行策略, 将所有相关的模型都加载到所有进程中
- 代码执行: 在计算Reward时, 生成的代码通常在本地直接执行, 缺乏完善的隔离机制
### 挑战
- PPO的训练极其昂贵, 他需要同时维护4个LLM, 极易造成OOM
- 代码执行环境需要做隔离, 否则存在安全风险, 且穿行执行效率比较低
### 贡献
>[!核心论点] 
> - 将算法逻辑和物理资源解耦, 并将模型训练和代码执行物理隔离
> - 标准的单体式PPO训练架构无法扩展, 需要使用分布式计算框架对模型进行分离

## Method
### 基于分布式计算的资源解耦
- 动机: 传统的Pytorch的分布式训练, 如DDP, 通常假设模型是静态绑定的, 但在PPO中, Actor需要更新梯度, 而打分模型只需要推理, 把他们绑在一个GPU上, 不仅显存不够, 计算负载也不均衡

- 工作机制
	- ### WorkerGroup
		- 封装了一组Worker, 每个worker可以是一个GPU
		- 那么每个Worker上作为执行的基本单位, 他们装配了vLLM等框架用于运行模型
		- Ray通过RPC控制所有的WorkerGroup, 相当于一个C/S服务器, 然后每个Worker只负责具体的运算, Ray的主节点则只负责资源调配
	- ### ResourcePool
		- 物理上的资源池, 虚拟化了一组GPU设备
		- 当ResourcePool应用与具体的WorkerGroup时, 相应的LLM计算会被映射到所分配的GPU上
- 流程
	- 定义一个Actor, 并为其分配ResourcePool A(如GPU0-3)
	- 定义一个Critic, 并为其分配ResourcePool B
	- Driver程序(如PPO算法) 通过远程过程调用协助这些WorkerGroup, 数据在不同的GPU组之间通过网络传输, 而不是显存拷贝

### 沙箱服务
- 技术动机: 生成的代码可能包含rm -rf / 等恶意指令, 同时成千上万条代码需要并发测试, 串行测试太慢
- 工作机制
	- 隔离性: 使用无特权的Docker容器, 每个容易内并发限度为1, 确保资源独占且互不干扰
	- 数据集管理: 为了减少网络开销, 测试用例预先存储在沙箱中, 训练时只需要发送题目ID和生成的代码, 无需传输巨大的测试数据
